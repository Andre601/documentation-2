Components

This tutorial is part of Iconify Icon Finder packages tutorial.

Components package is a set of reusable Svelte components.

Main entry points are src/wrapper.ts for Wrapper and src/ui/Container.svelte for container component.

Wrapper and Container

Wrapper class creates instance of container component and renders it inside DOM node that you specify. Then it handles events, it has functions to set and get config, route, selected icon, customisations.

Container component is needed to be passed as parameter to Wrapper instance. This allows you to swap container with another component, for example, one that renders your custom header, footer or navigation.

Example

Basic example:

// Import Wrapper class and container component
import { Wrapper } from '@iconify/search-components/lib/wrapper';
import Container from '@iconify/search-components/lib/ui/Container.svelte';

// DOM element to append icon finder to
const container = document.getElementById('container');

// Create instance
new Wrapper({
   container,
   component: Container,
   callback: (event) => {
       console.log('Event:', event);
   },
});

For more information about Wrapper class, take a look at Wrapper class documentation.

For more information about callback, take a look at Wrapper callback documentation.

Swapping components

You can swap components with custom components during build process.

To do that, put custom component in a directory within your implementation source that has structure similar to directory src of components package, add common.components.customDir to configuration and run build process. For more information about custom components, see custom components documentation.

Exports

There are several variables used in most components that are exported:

  • registry, Registry. This is an instance of Registry class. It never changes. It is used to retrieve static stuff that does not change, such as phrases, configuration, API providers.
  • selectedIcon, Icon | null. Selected icon as object.
  • route, PartialRoute. Current route, contains only parameters that are not default.

Additional exported variables that exist only in some components, but are still quite common:

  • iconCustomisations, IconCustomisations. List of icon customisations, such as color, dimensions, transformations. See src/misc/customisations.ts for type description.
  • blocks. List of blocks, specific to view. Used in view specific pages, so type is always known.
  • block. Block data, used in blocks. Type is specific to block.
  • name, string. Block name, used in blocks.

Actions, Routes

Actions in components, such as clicking page in pagination block, are handled by Router. So are routes.

Router is static, it can be retrieved from Registry instance.

Example:

registry.router.action('pagination', value);
registry.route = {
   type: 'collection',
   params: {
       prefix: 'mdi',
   },
   parent: {
       type: 'collections',
   },
};

Other stuff

Everything else is specific to component. See various components for code examples.

TypeScript

If you are using TypeScript, you can import types for various objects directly from Core by referencing @iconify/search-core package. Do not import files from relative directory, use package names instead. Lerna will handle all links to packages.

If you are experiencing errors in VSCode, such as VSCode telling you that imported type does not exist, run npm run link in root directory to fix symbolic links. If that does not help, edit import line once (cut and paste it to the same place, so code is not changed) to force VSCode to refresh import. It should fix VSCode glitch that sometimes happens when working with monorepo.
